Základní cíle kryptografie

Kryptografie řeší čtyři bezpečnostní vlastnosti dat: důvěrnost (confidentiality), integritu (data integrity), autentizaci (authentication — prokázání totožnosti odesílatele a původu dat) a nepopiratelnost (non-repudiation — nelze popřít odeslání). Kryptografická primitiva jsou základní stavební bloky; kombinací vznikají kryptografické protokoly (TLS, SSH, Kerberos, Blockchain).

Symetrická kryptografie

Obě strany používají stejný tajný klíč — jak pro šifrování, tak pro dešifrování. Výhody: vysoká rychlost, bezpečné a prověřené algoritmy. Nevýhoda: nutnost bezpečného kanálu pro distribuci klíče; nelze rozlišit, kdo ze dvou stran zprávu vytvořil.

Proudové vs. blokové šifry

Proudová šifra (Stream cipher)
Šifruje bit po bitu (nebo byte po byte) kombinací s pseudonáhodným proudem (keystream). Příklady: RC4 (zastaralý, zranitelný), ChaCha20 (moderní, používá TLS 1.3). Výhoda: rychlý, vhodný pro neznámou délku zprávy (streamy). Klíčové požadavky: nikdy znovu nepoužít stejný nonce se stejným klíčem.
Bloková šifra (Block cipher)
Šifruje pevné bloky (64 bit u 3DES, 128 bit u AES). Délka klíče: DES 56 bitů (prolomeno 1998, nepoužívat), 3DES 112 bitů (dosluhující), AES-128/192/256 (standard). NIST doporučuje minimální sílu zabezpečení 128 bitů pro nové aplikace. Klíč AES se pravidelně obnovuje (doporučení: po 5 letech).

Asymetrická kryptografie

Každá entita má pár klíčů: veřejný (public key, lze sdílet) a soukromý (private key, musí zůstat tajný). Matematika staví na jednosměrných funkcích — snadno se počítají jedním směrem, prakticky neinvertovatelné bez dalšího tajemství.

RSA (Rivest–Shamir–Adleman)
Bezpečnost závisí na obtížnosti faktorizace velkých čísel (součin dvou prvočísel).
Šifrování: odesilatel šifruje veřejným klíčem příjemce → dešifrovat může pouze příjemce soukromým klíčem.
Podpis: podepisující hashe zprávy podepíše svým soukromým klíčem → kdokoli ověří veřejným klíčem.
Doporučená délka klíče: min. 2048 bitů (3072+ pro nové aplikace, síla 128 bitů). RSA je pomalé — nepoužívá se pro šifrování objemných dat (hybridní schéma).
ECC (Elliptic Curve Cryptography)
Bezpečnost závisí na obtížnosti diskrétního logaritmu na eliptické křivce. Klíče jsou výrazně kratší: 256bitový ECC klíč poskytuje sílu srovnatelnou s 3072bitovým RSA. Algoritmy: ECDSA (podpis), ECDH (výměna klíčů). Křivky: P-256 (secp256r1), P-384, X25519 (Curve25519, moderní). TLS 1.3 používá ECDHE pro forward secrecy.
Kvantová hrozba: Shorův algoritmus na kvantovém počítači rozkládá RSA a ECC. NIST v roce 2022 standardizoval post-kvantové algoritmy (CRYSTALS-Kyber pro šifrování, CRYSTALS-Dilithium pro podpis). Symetrická kryptografie je méně zasažena — kvantové počítače snižují efektivní délku klíče na polovinu, takže AES-256 zůstává silné.

Hybridní šifrování (krok za krokem)

Asymetrická kryptografie je pomalá pro velká data. Řešení je hybridní schéma: asymetricky se přenese symetrický klíč (session key), veškerá data pak šifruje rychlá symetrická šifra.

1
Odeslatel generuje náhodný session key (např. 256bitový AES klíč) pro tuto konkrétní relaci.
2
Session key se zašifruje veřejným klíčem příjemce (RSA nebo ECDH). Vzniká „digital envelope" (obálka).
3
Zpráva se zašifruje session keym (AES-256-GCM) — rychlé, bez ohledu na velikost dat.
4
Příjemce rozšifruje obálku svým soukromým klíčem → získá session key → dešifruje zprávu.
5
Session key se po relaci zahodí — i při kompromitaci soukromého klíče v budoucnu nelze zprávy dešifrovat (forward secrecy, pokud klíče jsou efemérní).

Příklady: TLS (ECDHE + AES-GCM), PGP/GPG (šifrování e-mailů), S/MIME.

Hašovací funkce

Hash je jednosměrná funkce h(m) = H, která z libovolně dlouhé zprávy vytvoří pevnou hash hodnotu (digest). Tři klíčové vlastnosti:

  • One-way (preimage resistance) — z H nelze zpětně získat m.
  • Collision resistance — nelze nalézt dvě různé zprávy m₁ ≠ m₂ se stejným H. Kolize kompromituje podpisy a certifikáty.
  • Avalanche effect — minimální změna vstupu (1 bit) způsobí dramatickou změnu hashe (≈50 % bitů se změní).

MAC vs. digitální podpis

MAC (Message Authentication Code)
Hash zprávy kombinovaný se sdíleným tajným klíčem. Příklady: HMAC-SHA256, CMAC (pro blokové šifry).
Zajišťuje: integritu + autentizaci zdroje.
Nezajišťuje: nepopiratelnost — MAC může vytvořit i příjemce (oba znají klíč).
Použití: API authentication (HMAC v hlavičkách), ověření integrity souborů mezi dvěma stranami se sdíleným tajemstvím.
Digitální podpis (asymetrický)
Hash zprávy zašifrovaný soukromým klíčem podepisujícího. Ověření veřejným klíčem.
Zajišťuje: integritu + autentizaci + nepopiratelnost.
Klíčový rozdíl: pouze vlastník soukromého klíče mohl podpis vytvořit → nepopiratelnost (non-repudiation).
Algoritmy: RSA-PKCS1, ECDSA, EdDSA (Ed25519). Nutnost PKI nebo jiného modelu důvěry pro distribuci veřejných klíčů.

PKI — Public Key Infrastructure

PKI je soubor politik, procedur a technologií pro správu digitálních certifikátů a ověření vazby veřejný klíč ↔ identita. Certifikát X.509 je datová struktura obsahující: identifikační údaje (Subject/Issuer v Distinguished Name), veřejný klíč, dobu platnosti, sériové číslo a digitální podpis vydávající CA.

Hierarchie PKI

Root CA (Kořenová CA)
Vrchol hierarchie. Self-signed certifikát předinstalovaný v OS/prohlížeči (Mozilla NSS, Windows, macOS trust stores). Root CA by měla být offline — klíč uložen v HSM, fyzicky zabezpečen. Kompromitace Root CA = kompromitace celé hierarchie.
Intermediate CA (Mezilehlá CA)
Certifikát vydán Root CA. Online, vydává koncové certifikáty. Izoluje Root CA — i při kompromitaci Intermediate CA lze Root CA použít k vydání nové. Příklady: Let's Encrypt R3, R10 (podepsány ISRG Root X1).
Koncový certifikát (End-entity)
Certifikát serveru (TLS), osoby (S/MIME, kvalifikovaný podpis) nebo kódu (code signing). Obsahuje rozšíření: Subject Alternative Name (SAN — domény), Key Usage (šifrování vs. podpis), Extended Key Usage (TLS server auth, email, …).

Odvolávání certifikátů: CRL vs. OCSP

CRL (Certificate Revocation List)
Periodicky vydávaný seznam odvolaných sériových čísel (podepsán CA). Nevýhody: velký soubor (stovky KB), periodická aktualizace = zpoždění, klientská strana musí stáhnout celý seznam. Dnes nahrazován OCSP.
OCSP (Online Certificate Status Protocol)
Online dotaz na konkrétní certifikát (požadavek s hash certifikátu → odpověď „good/revoked/unknown"). OCSP Stapling: server si sám stapl (předem získá a k TLS handshaku přiloží) OCSP odpověď → klient nemusí kontaktovat OCSP responder → lepší soukromí a rychlost.

Kvalifikované certifikáty v ČR — PostSignum a I.CA

PostSignum (Česká pošta) a I.CA (První certifikační autorita, a.s.) jsou kvalifikované certifikační autority dle eIDAS nařízení EU č. 910/2014. Vydávají kvalifikované certifikáty fyzickým osobám a organizacím. Kvalifikovaný elektronický podpis (QES) má právní účinky rovnocenné vlastnoručnímu podpisu. Ověření identity před vydáním: fyzicky na pobočce nebo prostřednictvím NIA (Národní identitní autorita). I.CA je přítomna i v korporátním segmentu (certifikáty pro SSL servery, code signing).

eIDAS (nařízení EU 910/2014): Definuje 4 úrovně elektronického podpisu — prostý elektronický podpis (PES), zaručený (AdES), uznaný/zdokonalený (AdES + kvalifikovaný certifikát) a kvalifikovaný (QES — kvalifikovaný certifikát + kvalifikovaný prostředek pro vytváření podpisu, např. čipová karta nebo USB token). QES je právně nejsilnější a cross-border uznávaný v EU.

Faktory autentizace

Autentizace pracuje s ověřovači (verifikátory) — uloženými tajemstvími nebo vlastnostmi, které prokazují identitu. Faktory se dělí do tří kategorií:

Znalost — Something You Know
Heslo, PIN, odpověď na bezpečnostní otázku. Nevýhody: zapomínání, sdílitelnost, útoky hrubou silou, phishing. Heslo musí být uloženo jako hash (ne plaintext) se solí.
Vlastnictví — Something You Have
Hardware token (TOTP generátor — Google Authenticator), SMS OTP, smart card, FIDO2 bezpečnostní klíč (YubiKey). SMS je nejslabší — SIM swapping útok. FIDO2 hardware klíč je nejsilnější.
Biometrie — Something You Are
Otisk prstu, sken obličeje (Face ID), sken duhovky, hlas. Výhoda: nelze zapomenout. Nevýhody: biometrie se nedá změnit po kompromitaci, falešná přijetí (FAR) vs. falešná odmítnutí (FRR).

Bezpečné hashování hesel: bcrypt vs. Argon2

bcrypt a Argon2 jsou záměrně pomalé hashovací funkce navržené pro ukládání hesel. Na rozdíl od SHA-256 (navržen pro rychlost) jsou odolné vůči GPU/ASIC útokům díky nastavitelnému work factor (cost parametr). Argon2 (vítěz Password Hashing Competition 2015) nabízí navíc paměťovou náročnost (memory-hard) a paralelizaci. Doporučení NIST SP 800-63b: min. 8 znaků, neomezovat složitost, kontrolovat proti breached passwords, solení + iterovaný hash.

MFA — Multi-Factor Authentication

MFA (2FA) kombinuje alespoň dva různé faktory autentizace. Blokuje odhadem 99,9 % automatizovaných útoků (dle Microsoft Security). Implementace seřazeny od nejméně po nejbezpečnější:

  1. SMS OTP — nejrozšířenější, ale nejslabší (SIM swapping, SS7 útoky). Stále výrazně lepší než jen heslo.
  2. TOTP (Time-based OTP) — standardizováno v RFC 6238; Google Authenticator, Authy. 6místný kód platí 30 sekund. Sdílené tajemství uloženo v aplikaci.
  3. Push Notification — Duo Security, Microsoft Authenticator. Ohroženo MFA fatigue (útočník záplava notifikací).
  4. FIDO2 / WebAuthn — hardware klíč (YubiKey) nebo device-bound credential (Windows Hello, Face ID). Phishing-resistant — odpověď kryptograficky vázána na origin (doménu). Nejbezpečnější varianta.
FIDO2 / WebAuthn (W3C + FIDO Alliance): Klient generuje asymetrický pár klíčů vázaný na konkrétní origin (doménu). Soukromý klíč nikdy neopustí zařízení (uložen v TPM/Secure Enclave/hardware klíči). Ověření: server pošle challenge, klient podepíše soukromým klíčem. Phishing nefunguje — klíče jsou domain-specific.

IAM architektura

IAM (Identity and Access Management) je soubor procesů a technologií pro správu digitálních identit a řízení přístupu:

  • Identity Provider (IdP) — autentizuje uživatele a vydává tokeny/assertions (Active Directory, Azure AD/Entra ID, Okta, Auth0).
  • Service Provider (SP) — aplikace, která deleguje autentizaci na IdP a přijímá tokeny.
  • Provisioning — automatizované zakládání účtů při nástupu (onboarding) prostřednictvím HR systému → IdP → aplikace. Standard SCIM (System for Cross-domain Identity Management) pro automatizaci.
  • Deprovisioning — okamžité rušení přístupů při odchodu zaměstnance (offboarding). Jedno z nejrizikovějších míst: orphaned accounts.
  • Directory Services — Microsoft Active Directory (LDAP + Kerberos), Azure AD/Entra ID (cloud), LDAP (OpenLDAP).

RBAC — Role-Based Access Control

Uživatelé mají přiřazeny role; rolím jsou přiřazena oprávnění (operace na objektech). Uživatel → Role → Oprávnění → Objekt. Klíčové vlastnosti: role mohou být hierarchické, subjekt může mít více rolí (práva se sčítají), lze definovat separation of duties (SOD — neslučitelné role).

RBAC — příklad (WordPress)
Role: administrator, editor, author, contributor, subscriber. Operace v oprávnění: edit_published_posts, upload_files, publish_posts, delete_posts, read. Role author = publish_posts + edit_posts + upload_files + delete_posts + read. Vlastnictví: author může editovat pouze vlastní příspěvky.
RBAC vs. ABAC
RBAC: jednoduchost, výkon, snadno auditovatelný. Vhodný pro stabilní role.
ABAC: granulárnost, flexibilita, kontext (čas, IP, zařízení). Složitější implementace, pomalejší vyhodnocení. Kombinace běžná: role je jedním z atributů v ABAC politice.

ABAC — Attribute-Based Access Control

Přístup se vyhodnocuje na základě atributů subjektu (uživatele), atributů objektu (zdroje), atributů prostředí (čas, IP, způsob přihlášení) a atributů operace. Politiky se definují pomocí jazyka XACML nebo podobných.

Příklady ABAC politik z přednášky: „Dan může editovat článek, pokud je jeho autorem a článek není publikován." | „Dan má přístup pouze ze školní IP adresy." | „Dan se přihlásil přes 2FA → může editovat; jinak pouze číst." | „Soubor stažený z internetu → otevřít jen pro čtení."

PAM — Privileged Access Management

PAM chrání privilegované účty (admin, root, service accounts) — nejatraktivnější cíle útočníků. Klíčové principy:

  • Just-in-Time (JIT) Access — privilegovaný přístup udělen jen na nezbytnou dobu (minuty/hodiny), pak automaticky odvolán. Eliminuje trvalá admin oprávnění.
  • Session Recording — nahrávání video/textového záznamu privilegovaných relací pro audit a forenzní analýzu.
  • Password Vaulting — hesla admin účtů uložena v šifrovaném trezoru; aplikace je čerpají automaticky bez znalosti uživatele (CyberArk, BeyondTrust, HashiCorp Vault).
  • Dual Control — kritické operace vyžadují schválení druhé osoby (four-eyes principle).
  • Least Privilege — ani administrátor nemá více práv, než je pro danou úlohu nezbytné.

SSO protokoly: SAML 2.0, OAuth 2.0, OpenID Connect

Single Sign-On (SSO) odděluje autentizaci od aplikace. Uživatel se autentizuje jednou u IdP; SP přijme assertion/token. Výhody: uživatel si nepamatuje mnoho hesel, kompromitace jedné aplikace nevyzradí heslo, snazší správa (onboarding/offboarding na jednom místě), snadné přidání MFA.

SAML 2.0 (Security Assertion Markup Language)

XML-based protokol, typicky webové SSO v podnikovém prostředí a federované identity (eduID, ADFS). SP přesměruje uživatele na IdP s SAML AuthnRequest (šifrovaný veřejným klíčem IdP). Uživatel se přihlásí u IdP → IdP vydá SAML Assertion (XML dokument podepsaný soukromým klíčem IdP, šifrovaný veřejným klíčem SP). Assertion obsahuje: identitu uživatele, atributy (e-mail, skupiny), dobu platnosti. SP ověří podpis a přihlásí uživatele.

SAML flow (SP-initiated): Uživatel navštíví SP → SP přesměruje na IdP (AuthnRequest) → IdP autentizuje uživatele → IdP POST assertion na SP → SP ověří, přihlásí, přesměruje. Použití na VŠE: Office 365, vydávání certifikátů CESNETu (tcs.cesnet.cz), eduroam hesla.

OAuth 2.0

Framework pro delegaci autorizace — ne autentizace. Aplikace (client) dostane access token k přístupu ke zdrojům uživatele (resource owner) na resource serveru, aniž by znala jeho heslo. Klíčové pojmy: Resource Owner (uživatel), Client (aplikace), Resource Server (API), Authorization Server (vydává tokeny). Typy grantů: Authorization Code (nejbezpečnější — pro web apps), Implicit (zastaralý), Client Credentials (machine-to-machine), Device Code (IoT). Tokeny jsou JSON, přenášeny přes TLS.

OpenID Connect (OIDC)

Nadstavba OAuth 2.0 pro autentizaci. Přidává ID token (JWT — JSON Web Token — obsahuje identitu uživatele: sub, iss, aud, iat, exp). Klient (Relying Party) získá ID token + access token; UserInfo endpoint vrátí atributy o uživateli. OIDC řeší problém, který OAuth 2.0 sám o sobě neřeší — prokázání identity uživatele (OAuth 2.0 řeší pouze delegaci práv).

Zero Trust — principy a architektura

Zero Trust (pojem poprvé 2010, Forrester/John Kindervag) je bezpečnostní paradigma pro prostředí, kde nelze definovat bezpečnou interní síť: cloud, BYOD, vzdálená práce, SaaS aplikace. Základní teze: „Never trust, always verify".

Klíčové myšlenky Zero Trust
1. Neexistuje bezpečná (důvěryhodná) síť — požadavek ze známé sítě nemusí být bezpečný.
2. Vše je dostupné z internetu, služby se přesouvají do cloudu.
3. Více zdrojů důvěry: autentizace identity, místo, registrované zařízení, stav zařízení (MDM compliance).
4. Předpokládat, že narušení je možné nebo že k němu již došlo.
5. Základ bezpečnosti: využívání standardů (silná autentizace, šifrování, least privilege).
NIST 800-207 — Zero Trust Architecture
Policy Engine (PE) — rozhoduje o přístupu na základě politiky, signálů ze všech zdrojů.
Policy Administrator (PA) — vykonává rozhodnutí PE, konfiguruje přístup.
Policy Enforcement Point (PEP) — vynucuje přístup k zdroji.
Principy: ověření každé identity (MFA, SSO), ověření každého zařízení (MDM), ochrana dat (klasifikace, šifrování), mikrosegmentace sítě, monitorování anomálií.

Mikrosegmentace a ZTNA

Mikrosegmentace (micro-segmentation) rozděluje síť do malých izolovaných segmentů (workload-level) — i v případě kompromitace jednoho segmentu útočník nemá lateral movement. Implementace: softwarově definované sítě (SDN), NSX, Kubernetes Network Policies.

ZTNA (Zero Trust Network Access) nahrazuje tradiční VPN. Místo přístupu k celé síti poskytuje přístup k konkrétní aplikaci, a to až po ověření identity + stavu zařízení. Příklady: Cloudflare Access, Zscaler ZPA, Cisco Duo. Na rozdíl od VPN: menší attack surface, granulárnost, žádné lateral movement.

SQL Injection — detail

SQL injection nastane, když aplikace sestavuje SQL dotaz zřetězením vstupů uživatele bez sanitizace. Útočník vloží SQL syntaxi, která změní logiku dotazu.

Příklad payloadu: Login formulář sestavuje dotaz:
SELECT * FROM users WHERE username='[input]' AND password='[input]'
Vstup: ' OR '1'='1' -- → dotaz se stane: ... WHERE username='' OR '1'='1' --' AND password='...' → vždy true, přihlásí útočníka jako první uživatel v DB.

  • Typy SQL injection: In-band (výsledek přímo v odpovědi), Blind (boolovský nebo časový — odpověď neobsahuje data, ale chování se mění), Out-of-band (exfiltrace DNS/HTTP kanálem).
  • Nástroje útoku: SQLmap, SQLninja, Mole — automatické testování a exploitace.
  • Obrana: (1) Prepared statements / parametrizované dotazy — vstup NIKDY není součástí SQL syntaxe. (2) ORM (Object-Relational Mapping) — abstrakce DB dotazů. (3) Whitelist validace vstupu. (4) WAF jako doplňkový filtr. (5) Minimální DB oprávnění pro aplikaci.

XSS — Cross-Site Scripting

Útočník vloží škodlivý JavaScript do webové stránky, který se vykoná v prohlížeči oběti. XSS umožňuje: krádež session cookies, keylogging, přesměrování, defacement, BeEF framework (browser exploitation).

Stored XSS
Škodlivý skript uložen v DB (komentáře, profily). Zobrazí se každému uživateli, který navštíví stránku. Nejnebezpečnější.
Reflected XSS
Skript v URL parametru, server ho vrátí v odpovědi bez sanitizace. Oběť musí kliknout na škodlivý odkaz. Časté ve vyhledávacích formulářích.
DOM-based XSS
Skript se nevykoná přes server, ale manipulací DOM na straně klienta (JavaScript čte URL/localStorage a vkládá do DOM bez sanitizace).

Obrana XSS: (1) Output encoding — escapování HTML entit (< → &lt;). (2) Content Security Policy (CSP) hlavička — definuje, odkud lze načítat skripty; zakazuje inline JavaScript; podporuje nonce pro legitimní inline skripty. (3) HttpOnly cookie — JavaScript nemá přístup k session cookie. (4) WAF filtrace. (5) Testování: Burp Suite, OWASP ZAP.

Content Security Policy (CSP): HTTP hlavička limitující zdroje, ze kterých browser může načítat obsah. Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-xyz' — zakazuje inline skripty a načítání skriptů z cizích domén. Ochrání i před clickjackingem (direktiva frame-ancestors 'none').

CSRF — Cross-Site Request Forgery

Útočník na cizím webu (example.cz) vytvoří požadavek k cílové aplikaci (insis.vse.cz). Pokud má oběť v prohlížeči platné session cookie pro insis.vse.cz, prohlížeč cookie automaticky přiloží → požadavek se provede jménem oběti.

Příklad: Útočník vloží <img src="https://bank.cz/transfer?to=attacker&amount=10000" width="1" height="1"> na škodlivou stránku. Oběť (přihlášená u banky) navštíví stránku → prohlížeč pošle GET požadavek s cookie → převod proveden.

Obrana CSRF: (1) CSRF Token — unikátní náhodný token vložen do formuláře; server ověří. (2) SameSite cookie atribut — Strict (cookie není odesláno z cizích webů vůbec), Lax (cookie odesláno jen při top-level navigaci, ne pro img), None (bez omezení — nutný Secure). (3) Same-Origin Policy — browser blokuje čtení odpovědí z jiného originu (ale požadavky stále odesílá!). (4) Referer/Origin header validace jako doplněk.

DoS / DDoS útoky

Volumetrické (Volumetric)
Zahlcení šířky pásma. UDP flood, ICMP flood, DNS amplification (útočník pošle malý dotaz s podvrženou zdrojovou IP → DNS server pošle velkou odpověď oběti). Amplification factor až 100×.
Protokolové (Protocol)
SYN flood — vyčerpání stavové tabulky firewallu/serveru. Ping of Death (historický). Smurf attack. Obrana: SYN cookies, stavový firewall, rate limiting.
Aplikační vrstva (L7)
HTTP flood — velký počet legitimně vypadajících HTTP požadavků. Slowloris — drží otevřená HTTP spojení zasíláním neúplných hlaviček. Těžko odlišitelný od legitimního provozu.

Obrana DDoS: (1) Scrubbing center — přesměrování provozu přes filtrovací centrum (Cloudflare, Akamai, AWS Shield), kde je škodlivý provoz odfiltrován. (2) Anycast routing — provoz distribuován do mnoha PoP (Points of Presence), kapacita útoku se rozmělní. (3) Rate limiting na L7 (WAF). (4) CDN a reverse proxy jako buffer.

Man-in-the-Middle (MitM)

Útočník se vloží mezi dvě komunikující strany — čte, upravuje nebo injektuje data, aniž to strany vědí.

  • ARP Poisoning / ARP Cache Poisoning: útočník posílá falešné ARP odpovědi do lokální sítě → ostatní zařízení si zapíší, že útočníkova MAC adresa odpovídá IP adrese brány → provoz jde přes útočníka. Nástroj: arpspoof, Ettercap.
  • DNS Spoofing / DNS Cache Poisoning: falešné DNS odpovědi → oběť přeložení domény na IP útočníka. Obrana: DNSSEC.
  • SSL Stripping (sslstrip): útočník stojí na cestě, degraduje HTTPS na HTTP → oběť komunikuje nešifrovaně (nevidí v adresní řádce https). Obrana: HSTS, HSTS Preloading.
  • Rogue WiFi Access Point: útočník vysílá stejné SSID jako legitimní AP → zařízení se připojí k falešnému AP (zejm. open sítě).

Obrana MitM: (1) HSTS (HTTP Strict Transport Security) — server deklaruje, že příští přístupy musí být přes HTTPS; prohlížeč odmítne HTTP a nelze obejít certifikátové varování. (2) Certificate Pinning — aplikace akceptuje pouze konkrétní certifikát/veřejný klíč. (3) CAA (DNS Certification Authority Authorization) — DNS záznam omezuje, které CA smí vydat certifikát pro doménu. (4) Certificate Transparency — monitorování certificate logů.

IDS vs. IPS — detekce a prevence průniků

IDS — Intrusion Detection System
Pasivní — monitoruje provoz a generuje alerty, ale nablokuje nic. Typ NIDS (Network IDS — analyzuje síťový provoz ze zrcadlícího portu) nebo HIDS (Host IDS — analyzuje logy, integritu souborů, registry systému). Nástroj: Snort, Suricata (NIDS), OSSEC (HIDS — analýza logů, kontrola integrity, detekce rootkitů, centralizovaný server + agenti).
IPS — Intrusion Prevention System
Aktivní — nasazeno inline (v toku provozu); detekuje škodlivou aktivitu a automaticky blokuje, resetuje spojení nebo quarantinuje. Kombinace IDS + akce (drop paketu, TCP RST, vypnutí služby). Riziko false positives — nesprávné blokování legitimního provozu. Proto musí být pečlivě kalibrováno.

Detekční metody IDS/IPS

Signature-based (Signaturová detekce)
Porovnání provozu s databází podpisů (signatures) známých útoků. Přesná pro známé hrozby, nulová false positive u dobře napsaných signatur. Nevýhoda: nedetekuje zero-day útoky (neznámý exploit bez signatury). Závisí na aktuálnosti databáze (denní aktualizace).
Anomaly-based (Detekce anomálií)
Stanoví baseline normálního chování; odchylky generují alert. Detekuje neznámé útoky (zero-day), nové malware. Nevýhoda: vyšší false positive rate, náročná kalibrace. Metody: statistické modely, machine learning, heuristiky. Příklady anomálií: skenování portů, neobvyklá komunikace do cizích zemí, velký objem dat v neobvyklou hodinu.

SIEM integrace: IDS/IPS generují alerty, které jsou agregována v SIEM systému (Splunk, Microsoft Sentinel, IBM QRadar). SIEM koreluje události z více zdrojů — jeden alert může být nedůležitý, ale kombinace (skenování portů + failed logins + neobvyklý outbound traffic) indikuje incident.

DMZ — Demilitarizovaná zóna

DMZ je síťová zóna oddělující internet od interní sítě, kde jsou umístěny veřejně přístupné servery (web servery, e-mailové servery, DNS, VPN concentrator). Architektura se dvěma firewally:

1
Vnější firewall (edge) — odděluje internet od DMZ. Povoluje jen specifické porty do DMZ (80/TCP, 443/TCP, 25/TCP SMTP). Blokuje vše ostatní z internetu.
2
DMZ zóna — web servery, reverzní proxy, e-mail gateway, API gateway, VPN server. Servery jsou „semi-důvěryhodné" — i při jejich kompromitaci útočník nesmí snadno přejít do interní sítě.
3
Vnitřní firewall — odděluje DMZ od interní sítě. Povoluje pouze nezbytné komunikace (DB server v interní síti přijímá dotazy jen z konkrétního web serveru v DMZ).
Co kam patří? DMZ: web servery, mail relay, reverzní proxy, SFTP servery, VPN concentrator, DNS resolver pro veřejnost. Interní síť: databázové servery, Active Directory, file servery, zálohovací systémy, HR/účetní aplikace, monitoring. Nikdy v DMZ: Domain Controller, interní databáze, zálohy.

VPN — typy a protokoly

VPN (Virtual Private Network) vytváří soukromé propojení přes veřejnou síť, typicky s autentizací a šifrováním.

Typy VPN dle použití

Remote Access VPN
Připojení jednotlivého zařízení (stanice, notebook) do firemní sítě. VPN server přidělí klientovi IP adresu a upraví routing. Veškerý nebo část provozu jde tunelem (split tunneling). Použití: práce z domova, přístup k interním systémům. Příklad: VŠE vpn.vse.cz (Cisco AnyConnect, OpenConnect).
Site-to-Site VPN
Propojení dvou podsítí (pobočky, datová centra) přes veřejný internet. VPN brána na každé straně tunelu — zařízení uvnitř sítí nemusí mít VPN klienta. Transparentní pro koncová zařízení. Příklad: VŠE propojuje rekreační areály (Dobronice) přes IPsec tunnel.
Public VPN
Komerční VPN od třetí strany (NordVPN, ExpressVPN). Cíl: anonymita, změna geografické polohy, obcházení omezení. Pozor: poskytovatel VPN vidí veškerou komunikaci. Nevhodné pro firemní použití. „Don't use VPN services" — EFF ssd.eff.org.

IPsec — transportní vs. tunelovací mód

Transportní mód (Transport Mode)
Šifruje pouze payload (obsah) paketu; originální IP hlavička zůstává viditelná. Point-to-point komunikace — ví se, kdo s kým komunikuje. Typicky s L2TP pro Remote Access VPN (L2TP/IPsec). Autentizace přes certifikáty nebo PSK (Pre-Shared Key). ESP protokol (IP číslo 50).
Tunelovací mód (Tunnel Mode)
Celý původní paket (včetně IP hlavičky) je zašifrován a zabalen do nového paketu s novou IP hlavičkou. Skrývá interní IP adresy. Standardní mód pro Site-to-Site VPN a Remote Access VPN. Vytváří SA (Security Association) přes port 500/UDP (IKE).

Síťová segmentace — VLANy a mikrosegmentace

Síťová segmentace rozděluje síť do izolovaných segmentů pro omezení útočníkova pohybu a zjednodušení pravidel.

  • VLAN (Virtual LAN) — logické oddělení na L2 (IEEE 802.1Q tagging). Každý segment má vlastní broadcast doménu. Komunikace mezi VLANy prochází routerem/firewallem → možnost filtrování. VŠE: ~120 podsítí, ~40 segmentů (zaměstnanci, studenti, eduroam, koleje, IP telefony, IP kamery, switche…).
  • Mikrosegmentace — granularnější oddělení na úrovni workloadů (jednotlivých serverů, kontejnerů). Softwarově definované (NSX, Kubernetes Network Policies). I při kompromitaci jednoho serveru útočník nemá přístup k ostatním.
  • Účel segmentace: omezit šíření útoků (lateral movement), zjednodušit firewall pravidla, oddělit kritická zařízení (IP telefony, kamery, tiskárny) od uživatelských stanic.

SIEM — Security Information and Event Management

SIEM agreguje, normalizuje a koreluje bezpečnostní logy z celé infrastruktury v reálném čase pro detekci incidentů.

SIEM architektura

1
Log Collection: agenti nebo syslog forwarding sbírají logy z firewallů, IDS/IPS, webových serverů, AD, cloudových služeb (Office 365), EDR, databází.
2

3

4

Penetrační testování — 5 fází

Penetrační test (pentest) je autorizované simulované testování bezpečnosti systému za účelem nalezení zranitelností dříve, než je najde skutečný útočník. Musí mít písemný souhlas vlastníka systému (bez souhlasu = trestný čin).

1
Reconnaissance (Průzkum): Pasivní (OSINT — LinkedIn, WHOIS, Google dorking, Shodan) a aktivní (ping sweep, banner grabbing, DNS enumeration). Cíl: zmapovat attack surface bez přímého kontaktu s cílovým systémem.
2
Scanning (Skenování): Aktivní skenování portů (Nmap), identifikace verzí softwaru a OS (fingerprinting), skenování zranitelností (Nessus, OpenVAS). Výsledek: seznam otevřených portů, služeb, OS verzí, potenciálních zranitelností.
3
Exploitation (Exploitace): Využití nalezených zranitelností k získání přístupu. Nástroje: Metasploit Framework, Burp Suite (web), SQLmap (SQL injection), Mimikatz (credentials). Scope musí být přísně dodržen (pouze autorizované systémy).
4
Post-Exploitation: Po průniku: privilege escalation (zvýšení oprávnění), lateral movement (šíření do dalších systémů), persistence (backdoor), data exfiltration (simulace krádeže dat). Prokázání reálného dopadu kompromitace.
5
Reporting: Komplexní zpráva s executive summary (pro management), technickými detaily (CVE reference, CVSS skóre, reprodukční kroky), rizikovou klasifikací a konkrétními doporučeními k nápravě (remediation). Klíčová část — bez dobrého reportu má pentest malou hodnotu.

Informační bezpečnost — CIA triáda a základní pojmy

Bezpečnost informací (information security) představuje ochranu informací ve všech jejich formách a po celý jejich životní cyklus. Tři pilíře (CIA triáda):

Důvěrnost (Confidentiality)
Informace přístupné pouze oprávněným osobám. Metody: fyzická bezpečnost, šifrování, autentizace, autorizace, DLP (Data Loss Prevention). Ohrožuje: neoprávněný přístup, únik dat, odposlech.
Integrita (Integrity)
Zabránit neoprávněným změnám dat při uložení, přenosu i zpracování. Metody: digitální podpisy, hashe, auditování operací. Ohrožuje: úprava dat útočníkem, malware, chyby.
Dostupnost (Availability)
Dostupnost systémů a informací pro oprávněné uživatele. Metody: redundance, zálohy, BCP/DRP, aktualizace, řízení prostředí (napájení, chlazení). Ohrožuje: DDoS, ransomware, hardwarové selhání.
Kybernetická bezpečnost vs. informační bezpečnost: Bezpečnost informací chrání i analogové informace a před fyzickými hrozbami (požár, krádež zařízení). Kybernetická bezpečnost může chránit i jiná aktiva než informace (semafory, zdravotní zařízení). Překrývají se; NIS2 (zákon ZKB 181/2014 Sb. v ČR) hovoří o kybernetické bezpečnosti — gestor: NÚKIB.

Bezpečnostní politika — 3 úrovně

Bezpečnostní politika IS je soubor pravidel, směrnic a postupů chránících informační aktiva. Dělí se na tři hierarchické úrovně:

Strategická (Top-level)
Obecné prohlášení vedení o záměrech a cílech bezpečnosti. Schvaluje board / top management. Odkazuje na legislativu (ZKB, GDPR, NIS2) a normy (ISO 27001). Málo detailní, platí pro celou organizaci. Příklad: „Organizace se zavazuje chránit data zákazníků v souladu s GDPR."
Taktická (Issue-specific)
Politiky pro konkrétní oblasti: politika hesel (min. délka, MFA), politika zálohování (3-2-1), politika BYOD, politika acceptable use, politika incidentů. Schvaluje CISO. Závazná pro definované skupiny uživatelů.
Operativní (System-specific)
Konfigurace konkrétních systémů: bezpečnostní nastavení OS (hardening guide), firewall pravidla, ACL databáze, patch management plán. Nejdetailnější. Zodpovídá IT administrátor. Může mít formu technické dokumentace nebo runbooku.

Bezpečnostní dokumentace dle ISO 27001 a ZKB

ISO/IEC 27001:2022 — mezinárodní standard pro Information Security Management System (ISMS). Definuje 93 kontrol v Příloze A (reorganizovány do 4 témat: Organizational, People, Physical, Technological). Certifikace prokazuje ISMS auditované třetí stranou.

ZKB (Zákon o kybernetické bezpečnosti, 181/2014 Sb.) — český zákon definující povinnosti pro provozovatele kritické infrastruktury (KII — Kritická informační infrastruktura) a poskytovatele základních digitálních služeb. Transpozice NIS/NIS2 direktivy EU. Gestor: NÚKIB.

  • Politika ISMS — strategická politika bezpečnosti informací, cíle, závazky vedení.
  • Analýza rizik — identifikace aktiv, hrozeb, zranitelností, výpočet rizika (pravděpodobnost × dopad), plán zvládání rizik (risk treatment plan).
  • Prohlášení o aplikovatelnosti (SoA — Statement of Applicability) — které kontroly ISO 27001 Přílohy A jsou aplikovány a proč.
  • Bezpečnostní plán — konkrétní opatření, harmonogram, zodpovědné osoby.
  • Plán kontinuity (BCP — Business Continuity Plan) a obnovy (DRP — Disaster Recovery Plan).
  • Plán zvládání incidentů — identifikace, klasifikace, eskalace, zvládnutí, post-incident review.

Bezpečnostní modely přístupu

Formální bezpečnostní modely popisují pravidla pro řízení přístupu subjektů k objektům:

Bell-LaPadula Model (důvěrnost)
Zajišťuje důvěrnost utajovaných dat. Bezpečnostní úrovně (Přísně tajné / Tajné / Důvěrné / Veřejné). Pravidla: No Read Up (nelze číst z vyšší úrovně), No Write Down (nelze zapisovat na nižší úroveň). Vojenské a vládní systémy.
Biba Model (integrita)
Zajišťuje integritu dat. Opačná pravidla: No Read Down (nelze číst z nižší úrovně — kontaminace), No Write Up (nelze zapisovat na vyšší úroveň). Ochrana dat před neautorizovanými změnami. Použití v systémech vyžadujících integritu dat (vojenské povely, certifikace).

SIEM korelace a řetěz důkazů (chain of custody)

SIEM (Security Information and Event Management) ukládá události, analyzuje je a hlásí problémy. Klíčová funkce: korelace — kombinuje události z různých zdrojů pro detekci, které by jednotlivě nebyly patrné.

Příklady korelačních pravidel (z Office 365): přihlášení z anonymní IP (VPN/Tor), atypická cesta (přihlášení z geograficky vzdálených IP v krátkém čase, přičemž alespoň jedna je netypická), přihlášení z IP botnetu, rozesílání velkého množství odchozího SPAMu, neobvyklé vlastnosti přihlášení (neznámý prohlížeč, legacy protokol).

Řetěz důkazů (chain of custody) — pro forenzní analýzu a případné soudní řízení musí být prokázáno, že důkazy (logy, disk image, paměťový dump) nebyly od okamžiku zajištění modifikovány. Postup: pořízení hashe (MD5/SHA-256) ihned po zajištění, zdokumentování každého přístupu, práce s kopií (nikdy s originálem), chronologický záznam. Logy musí být tamper-proof — uloženy na médiu, do kterého nemůže útočník zapsat (write-once, WORM).

Vulnerability Management

Systematický cyklus identifikace, hodnocení, prioritizace a nápravy zranitelností v IT infrastruktuře:

1
Scan (Skenování): Pravidelné skenování sítě a systémů nástrojem pro vulnerability scanning (Nessus, Qualys, OpenVAS). Autentizovaný sken (s přihlašovacími údaji) odhalí více zranitelností než neautentizovaný.
2
Assess (Hodnocení): Výsledky skenu porovnány s CVE databází. Každé CVE dostane CVSS skóre. Kontextuální hodnocení — CVSS environmental metriky zohledňují prostředí organizace (kritičnost systému, dostupnost exploitu).
3
Prioritize (Prioritizace): Ne všechny zranitelnosti lze opravit okamžitě. Priority dle: CVSS skóre, dostupnosti exploitu (CISA KEV — Known Exploited Vulnerabilities), kritičnosti systému, dostupnosti patche, business dopadu.
4
Remediate (Náprava): Patch management, konfigurace změn (hardening), WAF pravidla jako dočasné opatření (virtual patching), vypnutí nepotřebné služby. Ověření úspěšnosti napravení následným skenem.

Cyklus se opakuje — nové CVE jsou denně zveřejňovány, prostředí se mění. Doporučení: skenovat minimálně měsíčně (kritické systémy týdně), patch Critical/High do 30 dnů (CISA: KEV do 2 týdnů).

Patch Management

Patch management je proces řízení a aplikace záplat (oprav) softwaru a firmwaru. Napojení na change management — každý patch prochází procesem: testování v dev/staging prostředí, schválení change advisory board (CAB), naplánované maintenance okno, rollback plán.

  • Emergency patches (zero-day): Kritické zranitelnosti s aktivním exploitací (např. Log4Shell) vyžadují urychlené záplatování mimo standardní cyklus. Musí projít zrychleným schvalovacím procesem, ale stále s dokumentací.
  • Patch Tuesday — Microsoft vydává záplaty druhé úterý každého měsíce. Organizace mají čas otestovat a nasadit ve standardním okně.
  • Automatizace: WSUS (Windows Server Update Services), SCCM/Intune pro Windows; Ansible, Puppet, Chef pro Linux servery; AWS Systems Manager Patch Manager pro cloud.
  • Inventarizace: Záplatovat lze jen to, co je v inventáři — asset management je předpokladem patch managementu.

Strategie zálohování

Záloha (backup) je kopie dat uložená na jiném médiu/místě pro obnovu po incidentu (ransomware, selhání hardware, lidská chyba, přírodní katastrofa).

Pravidlo 3-2-1

3× kopie dat
1× originální data + 2× zálohy. Tři kopie zaručují, že i při dvou selháních zůstane jedna použitelná.
2× různá média
Zálohy na různých typech médií (disk + páska, disk + cloud). Chrání před selháním specifickým pro jeden typ média.
1× vzdálená kopie
Alespoň jedna záloha offsite (jiná budova, datové centrum, cloud). Chrání před fyzickými katastrofami (požár, záplava, krádež hardware).

Rozšíření: 3-2-1-1-0 — navíc 1× offline (odpojená od sítě — chráněno před ransomware) + 0 chyb při ověření (testování obnovy). Záloha bez pravidelného testování obnovy není záloha.

RTO a RPO

RTO — Recovery Time Objective
Maximální přijatelná doba výpadku systému po incidentu. Příklad: RTO = 4 hodiny → systém musí být obnoven do 4 hodin od incidentu. Ovlivňuje architekturu zálohovacího řešení (hot standby vs. cold standby).
RPO — Recovery Point Objective
Maximální přijatelné množství ztracených dat (čas). Příklad: RPO = 1 hodina → zálohy musí probíhat minimálně každou hodinu; data starší než 1 hodina mohou být ztracena. Ovlivňuje frekvenci zálohování.

RSAAsymetrická šifra (Rivest-Shamir-Adleman). Bezpečnost: faktorizace velkých čísel. Min. 2048 bitů. Podepisování = soukromý klíč; šifrování = veřejný klíč příjemce. Pomalé — používá se jen pro výměnu session key (hybridní schéma).
ECCElliptic Curve Cryptography — asymetrická kryptografie na eliptických křivkách. 256bitový ECC ≈ 3072bitový RSA. Algoritmy: ECDSA (podpis), ECDH/ECDHE (výměna klíčů). Křivky: P-256, X25519.
SHA-256 / SHA-3Kryptografická hashovací funkce. SHA-256 (SHA-2): 256bitový digest, standard v TLS/certifikátech. SHA-3 (Keccak): odlišná sponge konstrukce. Pro hesla: bcrypt nebo Argon2 (záměrně pomalé, memory-hard).
HMACHash-based Message Authentication Code — MAC kombinující klíč s hash funkcí (HMAC-SHA256). Zajišťuje integritu + autentizaci. Na rozdíl od digitálního podpisu nezajišťuje nepopiratelnost.
PKIPublic Key Infrastructure — hierarchie Root CA → Intermediate CA → End-entity certifikát X.509. Váže veřejný klíč na identitu. Odvolávání: CRL nebo OCSP (OCSP Stapling).
TLS 1.3RFC 8446 — kryptografický protokol L4. 1-RTT handshake (0-RTT volitelný), povinný PFS (ECDHE), šifrování handshaku, pouze 5 AEAD cipher suites. Odstraněno: RSA key exchange, MD5/SHA1, RC4, CBC.
PFS / Forward SecrecyPerfect Forward Secrecy — efemérní klíče (ECDHE) pro každou relaci. I při kompromitaci soukromého klíče serveru nelze dešifrovat minulé relace.
PostSignum / I.CAKvalifikované certifikační autority v ČR (Česká pošta / První certifikační autorita). Vydávají QES (kvalifikované elektronické podpisy) dle eIDAS. Právní účinky rovnocenné vlastnoručnímu podpisu.
eIDASNařízení EU 910/2014 — elektronická identifikace a důvěryhodné služby. Definuje úrovně el. podpisu: PES, AdES, QES. QES = kvalifikovaný certifikát + kvalifikovaný prostředek (čipová karta, HSM).
MFA / 2FAMulti-Factor Authentication — kombinace ≥ 2 faktorů (znalost + vlastnictví + biometrie). Blokuje 99,9 % automatizovaných útoků. TOTP (RFC 6238), FIDO2/WebAuthn (phishing-resistant).
FIDO2 / WebAuthnOtevřený standard pro passwordless a MFA autentizaci. Klient generuje pár klíčů vázaný na origin (doménu). Soukromý klíč nikdy neopustí zařízení (TPM/Secure Enclave). Phishing-resistant.
IAMIdentity and Access Management — správa identit, autentizace (IdP, SP), autorizace (RBAC/ABAC), provisioning/deprovisioning, directory services (AD, LDAP), audit.
RBACRole-Based Access Control — přístup přiřazen rolím; uživatel dostane roli. Hierarchické role, SOD (separation of duties). Výhoda: jednoduchost správy, auditovatelnost. Příklad: WordPress role administrator/editor/author.
ABACAttribute-Based Access Control — přístup řízen atributy subjektu, objektu, prostředí, operace. Flexibilnější než RBAC. Politiky v XACML. Také: PBAC, CBAC. Kombinace s RBAC (role je atributem).
PAMPrivileged Access Management — Just-in-Time access, session recording, password vaulting (CyberArk, BeyondTrust), dual control. Chrání privilegované účty (admin, root, service accounts).
SSOSingle Sign-On — přihlášení jednou, přístup k více aplikacím. Odděluje autentizaci od aplikace. Výhody: méně hesel, snazší offboarding, MFA na jednom místě.
SAML 2.0Security Assertion Markup Language — XML-based SSO protokol. IdP vydá XML Assertion podepsanou soukromým klíčem. SP ověří veřejným klíčem IdP. Použití: Office 365, eduID, eduroam na VŠE.
OAuth 2.0Framework pro delegaci autorizace k API. Resource Owner, Client, Resource Server, Authorization Server. Grant types: Authorization Code (nejbezpečnější), Client Credentials (M2M), Device Code (IoT). Přenáší Access Token (JWT nebo opaque).
OIDCOpenID Connect — nadstavba OAuth 2.0 pro autentizaci. Přidává ID Token (JWT s identitou: sub, iss, aud, iat, exp). Řeší to, co OAuth 2.0 sám neřeší — prokázání identity. „Přihlásit se přes Google/Microsoft".
Zero TrustBezpečnostní paradigma: „Never trust, always verify". Nevěřit ani vnitřní síti. Více zdrojů důvěry. NIST 800-207: Policy Engine / Administrator / Enforcement Point. Pilíře: Identity, Device, Network, App, Data.
ZTNAZero Trust Network Access — přístup k aplikaci (ne k celé síti). Náhrada VPN. Kontinuální ověřování identity + stavu zařízení. Cloudflare Access, Zscaler ZPA, Cisco Duo.
OWASP Top 102021: A01 Broken Access Control, A02 Cryptographic Failures, A03 Injection, A04 Insecure Design, A05 Security Misconfiguration, A06 Outdated Components, A07 Auth Failures, A08 Integrity Failures, A09 Logging Failures, A10 SSRF.
SQL InjectionVstup uživatele je interpretován jako SQL příkaz. Obrana: prepared statements (parametrizované dotazy), ORM, WAF. Automatické testování: SQLmap.
XSSCross-Site Scripting — škodlivý JavaScript vložen do stránky. Typy: Stored (v DB), Reflected (v URL), DOM-based. Obrana: output encoding, CSP hlavička, HttpOnly cookie.
CSRFCross-Site Request Forgery — útok ze třetí strany využívající session cookie oběti. Obrana: CSRF token, SameSite cookie atribut (Strict/Lax), Same-Origin Policy.
SSRFServer-Side Request Forgery — server vykonává HTTP požadavky na URL specifikované útočníkem. Přístup k interním službám, metadata API. Obrana: whitelist URL, blokování interních rozsahů.
DoS / DDoSDenial of Service / Distributed DoS — zahlcení cíle. Typy: volumetrické (UDP flood, DNS amplification), protokolové (SYN flood), aplikační (HTTP flood, Slowloris). Obrana: scrubbing center, anycast, rate limiting.
MitMMan-in-the-Middle — útočník vložen mezi komunikující strany. Techniky: ARP poisoning, DNS spoofing, SSL stripping, rogue AP. Obrana: TLS, HSTS, certificate pinning, DNSSEC.
HSTSHTTP Strict Transport Security — server deklaruje: veškerá komunikace musí být přes HTTPS. Prohlížeč odmítá HTTP a nelze obejít certifikátové varování. Chrání před SSL stripping. HSTS Preloading = vestavěno do prohlížečů.
CSPContent Security Policy — HTTP hlavička omezující zdroje skriptů, stylů, rámců. Chrání před XSS, clickjackingem. Direktiva: script-src, frame-ancestors, default-src. Nonce pro legitimní inline skripty.
WAFWeb Application Firewall — L7 firewall pro HTTP/HTTPS. Filtruje SQL injection, XSS, CSRF, path traversal. Implementace: Apache mod_security, Cloudflare WAF, AWS WAF. Nutná kalibrace, sledování false positives.
NGFWNext Generation Firewall — stavový FW + IPS inline + DPI + TLS/SSL inspekce (MitM na TLS) + App-ID + URL filtrování + User-ID + centralizovaná správa. Palo Alto, Fortinet, Cisco Firepower, Check Point.
IDSIntrusion Detection System — pasivní monitoring, generuje alerty. NIDS (síťový, ze zrcadlícího portu), HIDS (hostový, logy + integrita souborů). Detekce: signaturová + anomaly-based. Snort, Suricata, OSSEC.
IPSIntrusion Prevention System — aktivní inline blokování škodlivého provozu. IDS + automatická akce (drop paketu, TCP RST). Riziko false positives → nutná kalibrace.
DMZDemilitarized Zone — síťová zóna mezi vnějším firewally a vnitřní sítí. Veřejné servery (web, mail, DNS, VPN). Dvě-firewallová architektura (vnější FW → DMZ → vnitřní FW → interní síť).
VPNVirtual Private Network — soukromé propojení přes veřejnou síť. Typy: Remote Access, Site-to-Site, Public. Protokoly: PPTP (nebezpečný), IPsec (IKEv2/L2TP), OpenVPN/SSL VPN, WireGuard (moderní, nejrychlejší).
IPsecInternet Protocol Security — standard pro šifrování/autentizaci IP paketů. Transport Mode (payload šifrován, IP hlavička viditelná), Tunnel Mode (celý paket šifrován). ESP (protokol IP 50), IKE (UDP 500).
SIEMSecurity Information and Event Management — agregace, normalizace, korelace logů v reálném čase. Detekce incidentů. Splunk, Microsoft Sentinel, IBM QRadar, Elastic SIEM.
CVECommon Vulnerabilities and Exposures — standardizovaný identifikátor zranitelnosti (CVE-YYYY-NNNNN). Spravuje MITRE, databáze NVD (nvd.nist.gov). Příklad: CVE-2021-44228 = Log4Shell.
CVSSCommon Vulnerability Scoring System v3.1 — skóre 0–10. Skupiny: Base (0–10, neměnné vlastnosti), Temporal (dostupnost exploitu, patche), Environmental (kontext organizace). Critical ≥ 9.0.
PentestingPenetrační testování — autorizovaný simulovaný útok. 5 fází: Reconnaissance → Scanning → Exploitation → Post-Exploitation → Reporting. Výstup: zpráva s CVE/CVSS referencemi a doporučeními k nápravě.
Red / Blue / Purple TeamRed = útočník (simuluje APT). Blue = obránce (SOC, SIEM, incident response). Purple = kooperace — Red testuje techniku, Blue ověřuje detekci → iterativní zlepšení (MITRE ATT&CK coverage).
CIA triádaConfidentiality (důvěrnost) + Integrity (integrita) + Availability (dostupnost) — tři pilíře informační bezpečnosti. Základ ISO 27001, ISMS.
ISO 27001Mezinárodní standard pro ISMS (Information Security Management System). 93 kontrol v 4 tématech (Organizational, People, Physical, Technological). Certifikace auditovaná třetí stranou.
ZKB / NÚKIBZákon o kybernetické bezpečnosti 181/2014 Sb. (transpozice NIS/NIS2) — povinnosti pro KII/ZDS. Gestor: NÚKIB (Národní úřad pro kybernetickou a informační bezpečnost).
Bell-LaPadula / BibaBezpečnostní modely. Bell-LaPadula: důvěrnost (No Read Up, No Write Down). Biba: integrita (No Read Down, No Write Up). MAC modely pro klasifikaci dat.
3-2-1 / RPO / RTOStrategie zálohování: 3× kopie, 2× médium, 1× offsite. RPO = maximální ztráta dat (čas). RTO = maximální doba výpadku. 3-2-1-1-0 rozšíření: + 1× offline + 0 chyb při testování obnovy.
Phishing / Spear-phishingPhishing = hromadný podvod e-mailem. Spear-phishing = cílený (konkrétní osoba/org.). Vishing = telefonní. Pretexting = vymyšlený scénář. Obrana: security awareness training, MFA, simulované phishingové kampaně.
ARP poisoningÚtok na L2 — falešné ARP odpovědi → přesměrování provozu přes útočníka (MitM). Obrana: Dynamic ARP Inspection (DAI) na switchi, statické ARP záznamy.
MITRE ATT&CKZnalostní báze technik a taktik skutečných útočníků (TTPs). 14 taktik, 300+ technik. Používá Red Team pro plánování, Blue Team pro mapování detekčních mezer, SIEM pro korelační pravidla.
